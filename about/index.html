主要项目经历

  • 清理大师系列

    公司系列项目(合作开发完成)，包括杀毒，文件清理，微信 QQ 垃圾清理，通知清理等近 10 余个功能，主要负责电池检测模块，apk 文件清理模块，通知清理模块等 6 个功能模块的开发。

    由于多个 App 需要用到同个功能模块，因此通过封装通用模块来解决多项目复用的问题；通过导入共用模块，出现多个 App 同个功能下出现 Bug 的情况，只需修改公共模块代码，即可修复全部 App 的 Bug，更加方便。

    apk 等垃圾文件扫描过程，遇到不能彻底清查的问题，存在部分 apk 文件不能扫描到的问题，如果全盘进行扫描，则存在扫描时间过久的问题。通过对 apk 常用存储路径进行分析，通过正则表达式进行文件夹定位，同时采用递归方式，对文件夹进行扫描。

    大部分功能的查询过程相对比较久，为了让用户不觉得「无聊」，通过自定义 View 展示一些特殊的动画效果，通过动画可以展示查询的进度等。但是涉及动画很多情况都会造成 App 崩溃，通过 Monkey 对动画进行不断测试，尽可能的解决由于动画产生崩溃的问题。

  • 动态壁纸

    一款用于替换手机桌面壁纸和来电显示动画的 App，主要包括替换桌面壁纸，替换来电显示动画，来电铃声等功能。

    替换桌面壁纸，由于部分壁纸为视频壁纸，普通的壁纸加载方式无法满足视频播放的需求，通过对 Android Api 的分析，决定使用自定义 WallpaperService 的 Engine 进行加载视频播放器 MediaPlayer；解决了视频的加载问题，对内存和耗电的控制也是比较重要的环节，通过监听 Engine 的 Surface 可见性进行监听，当不可见时，将播放器暂停并释放，当可见时，重新加载视频文件，向用户展示视频壁纸。

    设置视频壁纸为来电展示，展示的时机可以通过来电广播进行监听，但是播放视频文件的载体是个难解决的问题，经过讨论研究，最终采用 WindowManager 加载 SurfaceView 作为播放器 MediaPlayer 的载体，播放的机制同动态壁纸。

    视频背景提取问题，目前比较方便的方案就是通过 FFmpeg 处理，但是有个缺点，FFmpeg 处理过程会占用较高的内存，当手机内存不足的时候，容易引发 OOM，通过对比 Android 自带 MediaExtractor 相关 Api 进行处理的结果，发现后者相对耗时增加了约 10% ~ 20%，内存方面降低了不少，同时，减少了 apk 文件近约 40% 的大小，最终放弃使用 FFmpeg 的方案。

  • 万年历

    一款集日历展示，黄历展示，黄历查询，日历提醒等功能的 App。

    开源的日历控件比较多，由于不支持装填自己的数据，所以就需要自定义日历控件，因为日历也可认为是一组横向列表展示，所以选择方案选定为 ViewPager 和 RecyvlerView中的一个 ，再考虑到 ViewPager 的预加载机制和 View 的管理机制更加适合当前情况，最终敲定使用 ViewPager进行二次开发。

    数据装载问题，因为日历的日期等基本数据基本固定，通过网络返回无疑消耗了不必要的流量，所以可通过加载本地数据解决，但是考虑到数据库的安全性和加密的难度，最终放弃使用数据库存放本地数据，而改为使用 so 库加载，so 库的加密相对更加方便，不容易被反编译，而且通过 JNI 的方式进行数据加载，同数据库进行加载的方式，时间上相差不大，在初始化完成后，so 库加载数据的耗时会更少。

    日历界面卡顿问题，通过将数据和界面分离测试，检查到是 JNI 查询数据的时间过久，造成界面刷新卡顿，修改查询的方式，由开始的单个日期查询，修改成日期列表查询，整体查询的速度加快近 20 倍，ViewPager 加载页面的数据，几乎可在 3ms 内查询完毕，同时将查询数据通过子线程查询，使得日历的滑动，翻页更加流畅，避免了界面卡顿，或者界面切换数据未加载成功等问题。

    日历提醒问题，目前市面上较多的方案通过写到 App 的数据库中进行提醒事件存储，再通过 App 对时间的监听进行提醒，但是考虑到「保活」，在高版本的手机上几乎无法实现的问题，所以上述方式很容易造成提醒事件无法提醒用户的情况，因此采用将提醒事件写到系统日历日程以及自带数据库，通过对系统日历日程修改的监听，修改数据库数据，同时在 App 启动的时候对系统日历数据和自带数据库数据进行对比，进行校准，保证用户写入的提醒事件能够准时准点提醒。
